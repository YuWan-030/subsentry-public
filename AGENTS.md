## 当前已落地记录

- 首次安装向导位于 `/install`，仅在当前数据库没有管理员账号时显示。
- 安装向导可创建第一个管理员账号、保存数据库配置到 `.env`、设置站点 URL、保存默认 Webhook。
- 已有至少一个管理员账号的旧部署不能被安装向导拦截。
- Linux 一键安装脚本位于 `scripts/install-linux.sh`，负责安装依赖、构建前端、写入 `.env`、配置 systemd 和 Nginx，并启动面板。
- 一键安装脚本默认使用 SQLite，启动后继续通过网页 `/install` 创建管理员，后续可在安装向导中切换数据库配置。
- 系统健康页仅管理员可见，导航位置在“系统设置”和“日志中心”之间。
- 高频自动任务状态应写入轻量任务状态，不要每 5 分钟刷入活动日志。
- 节点失败日志应按事件记录，持续离线状态留在系统健康页展示，不要刷屏日志中心。
- 客户资料已包含 `traffic_multiplier`，用于五倍速率扣减等流量倍率场景。
- 前端已包含 Web App Manifest 和 iOS 主屏幕 meta，移动端按网页 App 打开时仍建议 HTTPS 和同源部署。
- 当前前端主要页面包括：资产总览、客户管理、客户详情、个人中心、节点集群、用户管理、系统设置、系统健康、日志中心、安装向导。
- 当前后端 API 分组包括：`install`、`auth`、`dashboard`、`customers`、`settings`、`cron`、`logs`、`system`、`subscriptions`。
- 默认 Webhook 优先读取数据库设置 `default_webhook_url`，安装向导中填写的 Webhook 会被通知服务使用。
- 续费价格后端兜底规范化仍要保留，纯数字输入应统一为 `数字/月`。

## 项目记忆

### 当前目标
- 将 `SubSentry_app.py` 拆分为模块化后端。
- 后端改为 `FastAPI`，只提供 JSON API。
- 前后端彻底分离，前端后续迁移到 `React + Ant Design`。
- 逐步保留现有业务能力：客户管理、续费、统计、通知推送、系统设置。

### 现有项目特征
- 当前是单文件 `Flask` 应用，页面由 `templates/*.html` 提供。
- 数据库支持 `sqlite` / `mysql` 双模式。
- 业务曾集中在一个文件中，包含：
  - 数据库初始化与查询
  - 客户 CRUD
  - 续费与财务流水
  - 告警 Webhook 推送
  - 系统设置
  - 登录态与页面渲染

### 重构原则
- 后端只负责 API，不再渲染 HTML。
- 路由层保持薄，业务逻辑下沉到 `services`。
- 数据访问集中到 `db` / `repository` 层。
- 配置集中管理，敏感信息改为环境变量。
- 保持接口兼容性时，优先提供新 API，再逐步废弃旧接口。

### 建议目录结构
- `backend/app/main.py`：FastAPI 入口
- `backend/app/api/v1/`：接口路由
- `backend/app/core/`：配置、鉴权、依赖
- `backend/app/db/`：数据库连接与初始化
- `backend/app/models/`：数据模型
- `backend/app/schemas/`：请求/响应 schema
- `backend/app/services/`：业务逻辑
- `frontend/`：React + Ant Design 前端

### 接口分组建议
- `auth`：登录、退出、当前用户、个人中心
- `dashboard`：总览、月度收入
- `customers`：客户增删改查、续费、审计、Webhook 测试
- `settings`：节点、经理、通知模板、推送配置
- `cron`：告警检查与手动触发

### 迁移顺序
1. 先拆出 FastAPI 后端骨架。
2. 迁移数据库层与核心业务逻辑。
3. 保留原能力的 JSON 接口。
4. 新建 React 前端并接入 API。
5. 最后清理旧的 Flask 模板与页面路由。

### 代码约定
- 优先使用清晰、可维护的模块划分。
- 新增代码尽量避免把业务写进路由函数。
- 保持返回结构统一，错误信息尽量明确。
- 涉及文件编辑时，优先最小改动原则。

### 当前已落地行为
- 已有独立“用户管理”和“个人中心”概念：
  - `用户管理` 是管理员功能。
  - `个人中心` 是当前登录用户自助修改昵称和密码。
- 客户归属采用 `owner_username + manager 展示名` 双字段思路：
  - `owner_username` 用于权限与实际归属。
  - `manager` 用于界面展示与兼容旧数据。
- 非管理员新增客户时：
  - 不需要手动选择客户经理。
  - 后端必须默认把客户绑定给自己，不能依赖前端控制。
- 续费价格字段需要做统一兜底：
  - 如果用户只输入纯数字，如 `10`、`88.5`，系统默认规范化为 `10/月`、`88.5/月`。
  - 该规则应优先在后端保证，前端可辅助提示，但不能只做前端处理。

### 当前注意事项
- 项目中存在硬编码的敏感信息，后续要迁移到 `.env`。
- 当前模板页还在依赖 iframe 结构，后续会被 React 替代。
- 定时推送逻辑目前在后台线程中，后续建议改为独立任务调度。
- 移动端页面需要持续检查布局细节，特别是客户页顶部操作区与筛选区，避免按钮错位或换行遮挡。
